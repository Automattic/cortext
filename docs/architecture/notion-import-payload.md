# Draft Notion import payload shape

This is a technical sketch for a possible Notion importer. It is not implemented,
and it is not a stable API. It describes the normalized structure we would expect
from a Notion import before Cortext turns it into WordPress posts, fields, and meta.

Once assembled, the payload is self-contained: all collections, their schemas, and
their entries live in one object so relations can be resolved by ID without extra
network calls.

The current prototype client assembles the payload in two stages:

- `extractAll(key)` — one paginated `/search` returns every collection it can reach,
  with its schema inline. Cheap, runs on screen mount.
- `extractCollection(key, id)` — fetches the entries for one collection on demand,
  when the user picks it from the list. Defers the heavy I/O until intent is shown.

The final, written-out payload below is what the prototype would emit if both
stages were merged for export.

## Top-level envelope

```json
{
  "extracted_at": "2026-04-19T12:00:00Z",
  "collections": [ /* CollectionObject[] */ ]
}
```

## CollectionObject

```json
{
  "id":                  "3468bd85-3edc-803a-aa0d-eeb72fb159cc",
  "slug":                "venues",
  "title":               "Venues",
  "parent_database_id":  "c18e9347-bb55-4c69-b2cf-02e9e404b7aa",
  "fields":              [ /* FieldObject[] — the schema */ ],
  "entries":             [ /* EntryObject[] — the rows */ ]
}
```

`slug` is derived from `title` (lowercase, spaces → hyphens). It is used as the CPT
suffix in WordPress (`crtxt_{slug}`).

`id` is a Notion data source id. In the 2026-03-11 API the parent "database" is a
container and one database may hold multiple data sources; this importer treats each
data source as a standalone Cortext collection. `parent_database_id` is the
container's id and is preserved so siblings can be grouped at import time.

## A note on property IDs

Notion property IDs are short opaque tokens, not friendly names. Real-world examples:
`pQqa`, `D{}j`, `=Ha=`, `>PdC`, `title`. The `title`-type property is conventionally
`"title"` in every database; the rest are generated by Notion and stable across renames.

Property IDs may contain characters that need URL-encoding (`>`, `=`, `{`, `}`, `[`,
`]`, `@`, `;`, …). The extractor normalises every property ID to its **decoded** form
so callers can cross-reference `fields[].id` and `entries[].values` keys without
per-site encoding logic.

## FieldObject

All fields share a base shape; some types carry extra config.

```json
{
  "id":   "notion-property-id",
  "name": "Status",
  "type": "status"
}
```

Type-specific additions:

| Type | Extra keys |
|------|-----------|
| `select` | `"options": [{ "id", "name", "color" }]` |
| `multi_select` | `"options": [{ "id", "name", "color" }]` |
| `status` | `"options": [{ "id", "name", "color" }]`, `"groups": [{ "id", "name", "color", "option_ids" }]` |
| `relation` | `"related_database_id": "<data-source-id>"` — points to one of `collections[].id` |
| `number` | `"format": "number" \| "dollar" \| …` |
| `formula` | `"expression": "<formula string>"` |
| `rollup` | `"relation_field": "<name>"`, `"rollup_field": "<name>"`, `"function": "<sum\|count\|…>"` |

Views are not in this payload at the moment. Notion saves per-database views
(table, calendar, board, …) with their own visibility/ordering of properties, but
the current prototype does not fetch them. They can be added back later if we want
to honour user-set column orders or filters at import time.

## EntryObject

```json
{
  "id":     "3468bd85-3edc-8095-bf2e-d03138c83d9a",
  "title":  "Hello, world",
  "values": {
    "<field-id>": <FieldValue>
  }
}
```

`title` is always the value of the `title`-type property. It maps to the WP post title
and is also mirrored in `values` under its field id.

`values` is keyed by **field id** (not name) to survive field renames.

## FieldValue types

| Notion type | Value shape | Notes |
|-------------|-------------|-------|
| `title` | `"string"` | Always present |
| `rich_text` | `"string"` | Plain text only |
| `number` | `42 \| null` | |
| `select` | `"Option name" \| null` | |
| `multi_select` | `["A", "B"]` | |
| `status` | `"Option name" \| null` | Treated like `select` |
| `date` | `"2026-04-16" \| null` | ISO 8601 date or datetime |
| `checkbox` | `true \| false` | |
| `url` | `"https://…" \| null` | |
| `email` | `"user@example.com" \| null` | |
| `phone_number` | `"+358…" \| null` | |
| `people` | `[{ "id": "<notion-user-id>", "name": "Miguel" }]` | Names may be unavailable |
| `relation` | `["<notion-entry-id>", …]` | IDs into `entries` of the related collection |
| `formula` | `<string \| number \| boolean \| null>` | Computed at export time |
| `rollup` | `<number \| array \| null>` | Computed at export time; skip on import |

Relations store **entry IDs**, not resolved objects. The importer resolves them by looking
up the target entry in the extracted payload, so there is no need for a separate lookup
pass.
