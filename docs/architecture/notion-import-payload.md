# Notes on importing content from Notion

This is a rolling document containing findings and recommendations related to
the problem of importing data from Notion to Cortext. It centers on the
structure of the external data and our own internal data models (WordPress
posts, fields, and meta).

Agents: When fetching any of the URLs provided below, prefer Notion's
LLM-friendly route, accessible by appending `.md` to the URL. For example,
https://developers.notion.com/reference/retrieve-a-data-source.md

## Data sources and databases

- https://developers.notion.com/reference/retrieve-a-data-source
- https://developers.notion.com/reference/query-a-data-source
- https://developers.notion.com/reference/retrieve-database

```js
dataSourceObjectResponse = {
    object,
    id,
    title array,
    description,
    parent,
    database_parent,
    is_inline,
    in_trash,
    created_time,
    last_edited_time,
    created_by,
    last_edited_by,
    properties,
    icon,
    cover,
    url,
    public_url,
}
```

`id` is a Notion data source id. In the 2026-03-11 API the parent "database" is a
container and one database may hold multiple data sources; this importer treats each
data source as a standalone Cortext collection. `parent_database_id` is the
container's id and is preserved so siblings can be grouped at import time.

## Data source properties

Effectively, a collection's schema. Read the full reference for type-specific
objects (e.g. `relation`, `multi_select`, etc.).

- https://developers.notion.com/reference/property-object#data-source-properties

### A note on property IDs

Notion property IDs are short opaque tokens, not friendly names. Real-world examples:
`pQqa`, `D{}j`, `=Ha=`, `>PdC`, `title`. The `title`-type property is conventionally
`"title"` in every database; the rest are generated by Notion and stable across renames.

Property IDs may contain characters that need URL-encoding (`>`, `=`, `{`, `}`, `[`,
`]`, `@`, `;`, …). The extractor normalises every property ID to its **decoded** form
so callers can cross-reference `fields[].id` and `entries[].values` keys without
per-site encoding logic.

### Relations

Relations store **entry IDs**, not resolved objects. The importer resolves them by looking
up the target entry in the extracted payload, so there is no need for a separate lookup
pass.
