# Importing from Notion

As of #275, Cortext can import content from Notion, provided the right _connection token_.

## Implementation highlights

The client stores the Notion token in `localStorage`, but all request to the Notion API are handled by the server, mediated by `Cortext\Notion\Client`. This mediator understands Notion's response properties `has_more` and [`next_cursor`](https://developers.notion.com/reference/post-search#response-next-cursor-one-of-0) in order to make serial requests until all paginated results have been fetched.

The REST controller, `Cortext\Rest\NotionController`, can be divided in two tasks:

- Discovering available databases in Notion (`/notion/collections`), which consists of querying the Notion API repeatedly for `data_source` objects.

- Importing a database by [fetching pages of a database](https://developers.notion.com/reference/query-a-data-source) in batches, creating Cortext rows with each new fetched batch. This is achieved using three endpoints (`notion/import/start`, `notion/import/tick`, `notion/import/finish`), explained below.

### Batching architecture

To minimise the complexities of server-side async jobs, **imports are orchestrated by the client**.

Once an import starts, the client calls the `start` endpoint. It will then keep calling the `tick` endpoint until the server signals that there are no more rows to import, at which point the client will call the `finish` endpoint.

Throughout this process, the client doesn't see any actual Notion data. It's merely acting as a scheduler for the server, instructing it to process the import the next batch of rows. In turn, each `tick` response contains a _count_ of rows processed so far, which we can feed into the UI to show progress. All this is encapsulated in `runImport`.

Meanwhile, the server relies on the three endpoints to 1) create, 2) update, then 3) delete a WordPress option — `cortext_notion_import_{$uuid}` — that represents the ongoing import job. This option contains all the necessary data for each new tick to query the Notion API with the right _cursor_ data.

## Notes on importing content from Notion

This is a rolling document containing findings and recommendations related to the problem of importing data from Notion to Cortext. It centers on the structure of the external data and our own internal data models (WordPress posts, fields, and meta).

With the introduction of `Cortext\Notion\Importer`, this section becomes increasingly less important, as knowledge is codified in actual code.

Agents: When fetching any of the URLs provided below, prefer Notion's LLM-friendly route, accessible by appending `.md` to the URL. For example, https://developers.notion.com/reference/retrieve-a-data-source.md

### Data sources and databases

- https://developers.notion.com/reference/retrieve-a-data-source
- https://developers.notion.com/reference/query-a-data-source
- https://developers.notion.com/reference/retrieve-database

```js
dataSourceObjectResponse = {
    object,
    id,
    title array,
    description,
    parent,
    database_parent,
    is_inline,
    in_trash,
    created_time,
    last_edited_time,
    created_by,
    last_edited_by,
    properties,
    icon,
    cover,
    url,
    public_url,
}
```

`id` is a Notion data source id. In the 2026-03-11 API the parent "database" is a container and one database may hold multiple data sources; this importer treats each data source as a standalone Cortext collection. `parent_database_id` is the container's id and is preserved so siblings can be grouped at import time.

### Data source properties

Effectively, a collection's schema. Read the full reference for type-specific objects (e.g. `relation`, `multi_select`, etc.).

- https://developers.notion.com/reference/property-object#data-source-properties

#### A note on property IDs

Notion property IDs are short opaque tokens, not friendly names. Real-world examples: `pQqa`, `D{}j`, `=Ha=`, `>PdC`, `title`. The `title`-type property is conventionally `"title"` in every database; the rest are generated by Notion and stable across renames.

Property IDs may contain characters that need URL-encoding (`>`, `=`, `{`, `}`, `[`, `]`, `@`, `;`, …). The extractor normalises every property ID to its **decoded** form so callers can cross-reference `fields[].id` and `entries[].values` keys without per-site encoding logic.

#### Relations

Relations store **entry IDs**, not resolved objects. The importer resolves them by looking up the target entry in the extracted payload, so there is no need for a separate lookup pass.
